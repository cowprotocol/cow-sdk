# EIP-7730 Clear-Signing Metadata for CoW Orders — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a single registry-ready EIP-7730 JSON metadata file describing the CoW Protocol `Order` EIP-712 message, covering all 11 production chains in one document.

**Architecture:** One static JSON file at `packages/order-signing/metadata/eip7730/eip712-cow-order.json` plus a short colocated README documenting how to validate and where to submit. No runtime/TS changes — the file is a deliverable for the Ledger clear-signing registry. Validation is manual (one `pip` tool invocation) and runs locally; not wired into the test/CI pipelines.

**Tech Stack:** JSON (EIP-7730 v1 schema). Manual validator: `erc7730` Python package (Ledger's reference linter). No new npm dependencies.

**Spec:** `docs/superpowers/specs/2026-05-12-eip7730-cow-orders-design.md`

````markdown
# EIP-7730 Clear-Signing Metadata

This directory contains [EIP-7730](https://eips.ethereum.org/EIPS/eip-7730) clear-signing metadata for CoW Protocol EIP-712 messages.

## Files

- `eip712-cow-order.json` — clear-signing metadata for the `Order` EIP-712 message (order placement). Covers all production chains where CoW Protocol is deployed.

## Validation

Before submitting to the Ledger clear-signing registry, validate the JSON locally:

```bash
pip install erc7730
erc7730 lint packages/order-signing/metadata/eip7730/eip712-cow-order.json
```

The `$schema` field in each JSON file also enables editor-side validation in VS Code / JetBrains IDEs.

## Submission

These files are designed for submission to the [Ledger clear-signing registry](https://github.com/LedgerHQ/clear-signing-erc7730-registry). Submission is a manual PR to that repository; this monorepo holds the source of truth.
````

## Task 3: Lint with Ledger's reference validator

The `erc7730` Python package is Ledger's reference linter. It catches schema violations the editor / `jq` checks won't. This task is the gate before opening a registry PR.

If `python3` / `pip` is not available in the dev environment, skip and document that fact in the README; the validation can be done by whoever submits the registry PR. Note that case at the end and the task is still considered complete.

**Files:**
- Modify: `packages/order-signing/metadata/eip7730/README.md` (only if `erc7730` finds issues — see Step 4)

- [ ] **Step 1: Install the validator in an isolated venv**

Run:
```bash
python3 -m venv /tmp/erc7730-venv && /tmp/erc7730-venv/bin/pip install --quiet erc7730
```

Expected: command completes with no errors. If `python3` is missing, install Python 3.10+ first (`brew install python@3.12` on macOS).

- [ ] **Step 2: Run the linter**

Run:
```bash
/tmp/erc7730-venv/bin/erc7730 lint packages/order-signing/metadata/eip7730/eip712-cow-order.json
```

Expected: linter exits 0 with no errors. Warnings about optional metadata (e.g., `deploymentDate` precision) are acceptable.

- [ ] **Step 3: Fix any errors the linter reports**

If the linter reports errors:
- For schema-shape errors (e.g. unknown property name, wrong type): edit `eip712-cow-order.json` to match what the linter expects. The error message names the path; consult <https://github.com/LedgerHQ/python-erc7730> for the schema definition.
- For semantic errors (e.g. an enum reference points at a missing key): edit the JSON to match the spec.
- Do **not** disable lint rules or skip the validator. The whole point of this task is gate-keeping pre-submission.

Rerun Step 2 after each fix until exit code is 0.
